# Kiwi — Sitemap & Product Structure

> Документ живий. Розділи додаються у міру просування від research → wireframes → design.

---

## Сутності продукту

> **Методологія:** сутність потрапляє в основний список тільки якщо за нею стоїть конкретний job з jtbd.md.
> Де поле лише припускається — [?]. Де сутність без job — секція «Під питанням».
> Джерела: personas.md, jtbd.md, research.md.

---

### 1. Проект

Центральна сутність продукту. Все, з чим взаємодіє користувач, так чи інакше пов'язане з Проектом.

**Поля:**

| Поле | Тип | Job | Примітка |
|------|-----|-----|---------|
| Назва | текст | RJ-1 | пошук по назві |
| Обкладинка | зображення | EJ-1 | перший візуальний фільтр · Інтерв'ю #1 |
| Екрани (2–5 шт.) | масив зображень | **RJ-2** | реальні скріни з файлу, не мокапи · ключове поле |
| Категорія | enum: Landing Page / Website / … | RJ-1 | груба навігація |
| Теги стилю/настрою | масив рядків | **RJ-1** | «мінімалістичний», «editorial», «нудний» · головний диференціатор |
| Інструмент | enum: Figma / Sketch / XD / … | RJ-1 | фільтр на каталозі |
| Ціна (разова) | число | RJ-4 | для не-підписників |
| Автор | посилання на Автора | EJ-1 | сигнал довіри |
| Дата додавання | дата | RJ-5 | для сортування «Нові» |
| Ліцензія | текст / enum | RJ-4 | комерційна, стандартна · підтверджено CLAUDE.md |
| Файл для завантаження | посилання | RJ-4 | сам файл (.fig / .sketch / …) |
| Editor's Pick мітка | boolean | EJ-3 [⚠️ гіпотеза] | кюраційна мітка від редакції |

**Job-підстава:** RJ-1, RJ-2, RJ-4, RJ-5, EJ-1
**Зв'язки:** Проект ↔ Автор · Проект ↔ Збережене · Проект ↔ Завантаження · Проект ↔ Покупка

---

### 2. Пошук

Механізм, через який користувач формулює запит і отримує список Проектів.

**Поля:**

| Поле | Тип | Job | Примітка |
|------|-----|-----|---------|
| Текстовий запит | рядок | **RJ-1** | «мінімалістичний», «editorial», «нудний» — мова дизайнера |
| Результати | список Проектів | RJ-1 | |

**Job-підстава:** RJ-1
**Зв'язки:** Пошук → Проект (повертає відфільтрований список)

> **Критично:** поточні конкуренти (UI8, Envato) мають пошук по категоріях, не по настрою. Це підтверджений gap · Інтерв'ю #1.

---

### 3. Фільтр

Набір параметрів для звуження списку Проектів без текстового запиту.

**Поля:**

| Поле | Тип | Job | Примітка |
|------|-----|-----|---------|
| Стиль / настрій | мультивибір тегів | **RJ-1** | головний фільтр |
| Інструмент | мультивибір enum | RJ-1 | Figma / Sketch / XD |
| Категорія | мультивибір enum | RJ-1 | Landing / Website |
| Ціна | діапазон або enum (безкоштовно / платно) | RJ-4 | |
| Дата додавання / сортування | enum: Нові / Популярні | RJ-5 | |

**Job-підстава:** RJ-1, RJ-4, RJ-5
**Зв'язки:** Фільтр → Проект (звужує список)

---

### 4. Збережене (Колекція)

Особиста бібліотека користувача — проекти, відкладені «на майбутнє».

**Поля:**

| Поле | Тип | Job | Примітка |
|------|-----|-----|---------|
| Список проектів | масив посилань | **RJ-3** | |
| Дата збереження | дата | RJ-3 | для сортування всередині |
| Назва колекції | рядок [?] | RJ-3 [⚠️ гіпотеза] | можливо потрібна тільки якщо є кілька колекцій |

**Job-підстава:** RJ-3
**Зв'язки:** Збережене ↔ Проект · Збережене ↔ Користувач

> **Примітка:** favorites як один список підтверджено Інтерв'ю #1. Кілька іменованих колекцій — [?], потребує валідації.

---

### 5. Завантаження

Подія отримання файлу — центральний момент цінності продукту.

**Поля:**

| Поле | Тип | Job | Примітка |
|------|-----|-----|---------|
| Проект | посилання | **RJ-4** | |
| Тип доступу | enum: підписка / разова покупка | RJ-4 | |
| Файл | посилання на файл | RJ-4 | |
| Дата | дата | — | для логів |
| Ліцензійна угода | boolean (прийнята) | RJ-4 | автоматично при скачуванні · CLAUDE.md |

**Job-підстава:** RJ-4
**Зв'язки:** Завантаження ↔ Проект · Завантаження ↔ Користувач · Завантаження ↔ Підписка · Завантаження ↔ Покупка

---

### 6. Підписка

Стан акаунту, що знімає friction при кожному завантаженні.

**Поля:**

| Поле | Тип | Job | Примітка |
|------|-----|-----|---------|
| Статус | enum: активна / неактивна | **RJ-4** | |
| Тип плану | enum: місячна / річна [?] | EJ-2 | |
| Дата початку | дата | EJ-2 | |
| Дата закінчення | дата | EJ-2 | |

**Job-підстава:** RJ-4, EJ-2
**Зв'язки:** Підписка ↔ Користувач · Підписка → Завантаження (розблоковує)

---

### 7. Покупка (разова оплата)

Альтернативний шлях отримання файлу без підписки.

**Поля:**

| Поле | Тип | Job | Примітка |
|------|-----|-----|---------|
| Проект | посилання | RJ-4 | |
| Сума | число | RJ-4 | ціна з поля Проекту |
| Дата | дата | — | |
| Статус оплати | enum | RJ-4 | |

**Job-підстава:** RJ-4 (для Мисливця на дедлайн — ⚠️ гіпотеза щодо цього сегменту)
**Зв'язки:** Покупка ↔ Проект · Покупка ↔ Користувач

> **Увага:** весь UX разової покупки будується під Мисливця на дедлайн — ⚠️ гіпотезу. RJ-4 для Куратора закривається підпискою.

---

### 8. Користувач (Акаунт)

Особа, яка взаємодіє з продуктом. Потрібен для Збереженого і Завантаження.

**Поля:**

| Поле | Тип | Job | Примітка |
|------|-----|-----|---------|
| Email | рядок | RJ-3, RJ-4 | логін |
| Пароль / auth | — | — | Supabase Auth |
| Підписка | посилання | RJ-4 | |
| Збережене | посилання | RJ-3 | |
| Історія завантажень | список | RJ-4 | |

**Job-підстава:** RJ-3 (потрібен акаунт щоб зберігати), RJ-4 (потрібен акаунт щоб скачувати)
**Зв'язки:** Користувач ↔ Підписка · Користувач ↔ Збережене · Користувач ↔ Завантаження · Користувач ↔ Покупка

---

### 9. Автор

Людина або команда, що завантажила Проект. З'являється як поле-сигнал довіри на Проекті.

**Поля:**

| Поле | Тип | Job | Примітка |
|------|-----|-----|---------|
| Ім'я | рядок | EJ-1 | сигнал довіри на картці |
| Аватар [?] | зображення | EJ-1 [слабко] | |
| Список проектів [?] | масив | — | жодного підтвердженого job «знайти за автором» |

**Job-підстава:** EJ-1 (слабко — як поле на Проекті, не як самостійний об'єкт навігації)
**Зв'язки:** Автор ↔ Проект

> **Примітка:** Автор як окрема сторінка або ціль навігації — [?] під питанням. Жоден підтверджений job не вимагає «перейти на профіль автора і подивитись інші роботи».

## Під питанням

Об'єкти, для яких jobs не знайдено або job є лише гіпотезою.

| Сутність | Чому під питанням | Який job міг би виправдати |
|----------|-------------------|---------------------------|
| Профіль автора (окрема сторінка) | Жоден підтверджений job не вимагає «переглянути всі роботи автора» | SJ-1 [⚠️ гіпотеза] |
| Командні колекції | Job SJ-2 (Арт-директор) — повністю гіпотетичний, 0 даних | SJ-2 [⚠️ гіпотеза] |
| Рейтинг / лайки від юзерів | Job EJ-3 — гіпотеза; конкуренти вже мають, диференціатором не стане | EJ-3 [⚠️ гіпотеза] |
| PRO-badge автора | Антипатерн з бенчмарку. Не закриває жоден job | — |
| Соціальна стрічка / follow автора | Тільки гіпотетичні SJ-1, SJ-2 | SJ-1, SJ-2 [⚠️ гіпотеза] |
| Кілька іменованих колекцій | RJ-3 підтверджено як «зберегти», але кілька колекцій — [?] | RJ-3 [розширення — не валідовано] |

## Екрани

> **Методологія:** екрани виведені з jobs (jtbd.md), не з конкурентів.
> Кожен екран — у дужках job, який він обслуговує.
> Немає job → [SIROTA].
> Стани (empty, error, loading) — не екрани, не включені.
> Глибина мінімальна — підрівні додамо свідомо на кроці 3.

## Навігація

> Навігація виведена з job-кластерів, не скопійована з конкурентів.
> Кожен пункт — вхід у кластер jobs, а не «бо так прийнято».

---

### 1. Глобальна навігація (4 пункти)

| # | Пункт | Job-кластер | Що людина хоче зробити |
|---|-------|-------------|------------------------|
| N1 | **Landing Page** | RJ-1 | «Шукаю готовий лендінг за стилем» — вхід у каталог з передфільтром категорії |
| N2 | **Website** | RJ-1 | «Шукаю готовий сайт за стилем» — те саме для вебсайтів |
| N3 | **Збережене ♥** | RJ-3 | «Повертаюсь до відкладеного» — особиста бібліотека |
| N4 | **Акаунт / Підписка** | RJ-4, EJ-2 | «Керую доступом до завантаження» — статус підписки, вхід/реєстрація |

**Логотип** (поза нумерацією) → повернення до каталогу з усіма категоріями [RJ-5]
Для Куратора, який приходить щодня подивитись нове — логотип = кнопка «додому».

**Чому саме ці 4, а не інші:**
- N1 і N2 — це категорії, а не просто «розділи». Вони скорочують RJ-1 на один крок: не потрібно відкривати каталог і тоді фільтрувати за типом. З часом сюди зайдуть Dashboard, Mobile, UI Kit — nav масштабується природно.
- N3 — підтверджена щоденна поведінка Куратора (Інтерв'ю #1). Без цього пункту в глобальній навігації шлях до C1 стає похованим.
- N4 — єдина точка керування підпискою (RJ-4). Якщо не видно — підписка «захована», як у UI8 (антипатерн з бенчмарку).
- Пошук і фільтри — **не** глобальна навігація. Вони контекстні: живуть на A1, зникають на B1.

---

### 2. Глибина тапів до main job

**Main job:** Куратор знаходить ресурс за стилем і отримує файл.
**Умова:** підписка вже є (підтверджений стан з Інтерв'ю #1).
**Лічимо:** переходи між екранами. Дії всередині екрана (фільтр, hover) — не тап.

```
Крок 0  →  A1 Каталог               (точка входу)
             │
Тап 1   →  B1 Сторінка проекту      (клік на картку)
             │
Тап 2   →  D1 Гейт · файл           (клік «Завантажити» → підписка активна → файл)

Разом: 2 тапи ✅
```

**Варіант без підписки (перший раз):**
```
Крок 0  →  A1 Каталог
Тап 1   →  B1 Сторінка проекту
Тап 2   →  D1 Гейт (немає підписки)
Тап 3   →  D2 Вибір плану
             └── Stripe (зовнішній)

Разом: 3 тапи до екрана оплати ✅
```

**Варіант збереження (RJ-3):**
```
Крок 0  →  A1 Каталог
Тап 1   →  іконка ♥ на картці       (дія на A1, без переходу)

Разом: 1 тап ✅
```

**Висновок:** глибина ≤ 3 для всіх primary-flows. Реструктуризація не потрібна.
Ключове рішення, що це забезпечує: каталог = точка входу (не окремий Home), реєстрація не блокує перегляд.
