## 1. Деталізація

Тільки структура, ієрархія і зони. Нічого, що belongs до візуального дизайну.

**Дозволено:**
- Блоки-зони (шапка, навігація, список карток, картка, форма, кнопка) як прямокутники з підписом ролі.
- Сіра шкала — від білого до чорного (наприклад: `#fff` фон, `#ccc` межі, `#999` заповнювачі, `#333` текст). Без відтінків, без акцентного кольору.
- Заповнювач замість зображення — сірий прямокутник з текстовою міткою на місці картинки (наприклад: `[екран проекту 1/3]`), не порожнє місце і не справжнє фото.

**Заборонено на цьому кроці:**
- Колір (окрім сірої шкали), бренд-палітра.
- Шрифти, вага, розмір як дизайн-рішення — типографіка за замовчуванням браузера.
- Логотип, іконки, ілюстрації, справжні зображення.
- Тіні, радіуси, градієнти, анімація.

Все з «заборонено» — не видалено назавжди, а свідомо відкладено (див. розділ 6).

## 2. Розмітка — семантичний HTML

Кожна зона — тег, що описує її роль, не `<div>` за замовчуванням.

| Роль зони | Тег | Приклад використання |
|-----------|-----|----------------------|
| Глобальна шапка сайту | `<header>` | Лого + N1–N4 глобальна навігація (sitemap.md §Навігація) |
| Навігаційний блок | `<nav>` | Список пунктів навігації, breadcrumb на B1 |
| Основна область екрана | `<main>` | Один `<main>` на сторінку |
| Змістовий блок усередині main | `<section>` | Блок фільтрів, блок «Екрани з файлу» на B1, форма на E1/E2 |
| Картка, що повторюється в списку | `<article>` | Картка проекту в каталозі, картка в «Збереженому» |
| Дія користувача | `<button>` | «Зберегти», «Завантажити», «Підписатись», «Спробувати ще» |
| Введення даних | `<form>` + `<input>`/`<label>` | Пошуковий рядок, форма реєстрації/входу, форма оплати |
| Список однорідних елементів | `<ul>`/`<li>` | Список карток, список тегів стилю |

`<div>` / `<span>` — тільки як layout-обгортка, коли жоден семантичний тег не підходить (напр. grid-контейнер для карток). Не за замовчуванням.

**Мінімальний скелет (ілюстрація патерну, не конкретний екран):**

```html
<header>
  <nav><!-- глобальна навігація --></nav>
</header>
<main>
  <section aria-label="Фільтри">
    <form><!-- пошук / фільтри --></form>
  </section>
  <section aria-label="Результати">
    <ul>
      <li><article><!-- картка --></article></li>
    </ul>
  </section>
</main>
```

## 4. Назви файлів

Базова сторінка екрана — успішний стан, без суфікса:

```
wireframes/<назва>.html
```

Кожен інший реальний стан з wireframes/_screens.md §Матриця станів — окрема сторінка з суфіксом:

```
wireframes/<назва>-empty.html
wireframes/<назва>-error.html
wireframes/<назва>-loading.html
```

`<назва>` — коротка англійська kebab-case назва екрана (не sitemap-ID на кшталт `a1`, а зрозуміла назва ролі екрана).

**Мапінг для 7 екранів wireframes/_screens.md:**

## 5. Стани — окрема сторінка на кожен

Один стан = одна сторінка, не перемикач/JS-стан всередині однієї сторінки. Причина: вайрфрейм — статичний артефакт для рев'ю і handoff, а не інтерактивний прототип.

**Правило незмінності структури:** усі сторінки одного екрана (`catalog.html`, `catalog-empty.html`, `catalog-error.html`, `catalog-loading.html`) використовують **той самий** набір зон і тегів (`header`, `nav`, `main`, ті самі `section`). Змінюється лише вміст усередині зони, де стан проявляється — решта сторінки (шапка, навігація, фільтри) залишається ідентичною.

Приклад для `listings`:
- `listings.html` — `<section aria-label="Результати">` містить `<ul>` з картками-`<article>`.
- `listings-empty.html` — та сама секція, замість `<ul>` — повідомлення «Нічого не знайшлось за цим запитом».
- `listings-error.html` — та сама секція, повідомлення «Помилка мережі» + кнопка «Спробувати ще раз».
- `listings-loading.html` — та сама секція, текстова/сіра позначка завантаження замість карток.

## 6. Що лишаємо на потім

Свідомо відкладено до наступних кроків (concept/ → tokens/ → design-system/ → components/ → handoff/ за CLAUDE.md):

- Колір і бренд-палітра
- Типографіка (шрифти, вага, розміри як рішення, не браузерний default)
- Тіні, радіуси, градієнти
- Іконки та ілюстрації
- Готовий UI / візуальний дизайн компонентів

Вайрфрейм відповідає тільки на питання «що є на екрані і в якій ієрархії», не «як це виглядає».

---

## 7. Заборонено: службові підписи й анотації

Вайрфрейм показує тільки те, що бачитиме користувач на екрані. Жодних службових написів для рев'ю чи трасування вимог у самій розмітці.

**Заборонено:**
- Підписи ролі зони («Зона: …», опис, що це за блок і яка його головна дія).
- Нотатки про позицію екрана в flow («Позиція в Main Job flow: …», посилання на кроки flows.md/sitemap.md всередині HTML).
- Будь-який інший текст, що існує для розробника/рев'ювера, а не для користувача продукту.

Трасування «екран → job → flow → sitemap» веде sitemap.md, flows.md, wireframes/_screens.md — не HTML-файл. HTML містить лише реальний UI-контент (§3).

## 8. Навігаційне дерево екранів

Кожна сторінка вайрфрейму містить зліва від макета `<nav class="wf-nav-tree">` — однакове дерево на **кожній** сторінці, без винятків.

**Структура (3 рівні, з відступом):**
```
Розділ (sitemap.md §Екрани · ГРУПА A–E)
  └── Екран (wireframes/_conventions.md §4 · Sitemap-ID + назва)
        └── Стан (wireframes/_screens.md §Матриця станів · тільки ✓-клітинки)
```

**Правила:**
- Розділи й екрани — текст, не посилання (не є окремою сторінкою).
- Лише стан — `<a>` з `href` на файл за мапінгом §4 (`<назва>.html`, `<назва>-empty.html`, …).
- Показуємо тільки ті стани, де в матриці §Матриця станів (_screens.md) стоїть ✓. «—» не додається і не вигадується.
- Область дії дерева — 7 екранів «Ядро MVP» з _screens.md (A1, B1, C1, D1, D2, E1, E2). D3, E3, S1–S4 — поза скоупом (_screens.md §Методологія відбору), у дерево не входять, доки не потрапляють в цей скоуп.
- Поточну сторінку позначає `aria-current="page"` на відповідному `<a>` — жирним/підкресленням, без кольору (§1).
- Сіра шкала, як і решта вайрфрейму — жодного акцентного кольору для дерева.
- Коли з'являється новий екран/стан у _screens.md — дерево оновлюється на **всіх** сторінках одночасно (це один і той самий блок, скопійований на кожну).
